# Review：Palantir FDE 深度解析：是一场营销游戏，还是软件工程的革命？

## 当前状态

待讨论。

这篇不是 ontology 定义文，也不是 AIP action 教程。它讨论的是 Palantir 的
`Forward Deployed Engineer`（FDE）到底是什么：是新的软件工程范式，还是把企业软件实施、
技术咨询、售前工程、客户成功和数据工程重新包装成一个更有吸引力的角色。

我建议把它作为“交付模型”材料来读，而不是作为产品能力证明来读。

## 先讲清楚：这篇到底举了什么例子

文章没有像前面的实战文那样给出一个完整业务 demo。它举的是一种典型企业软件交付场景：

```text
客户有复杂业务问题和混乱数据
-> 客户很难自己直接使用 Foundry / Gotham
-> Palantir 派 FDE 到客户现场
-> FDE 理解业务、接入数据、配置平台、设计工作流、处理权限和集成问题
-> 客户更快看到一个能运行的业务解决方案
```

更具体地说，文章把 FDE 的日常工作描述成：

- 设计、编写和测试工作流；
- 配置 Gotham / Foundry 的数据模型、权限和功能；
- 与客户大量沟通；
- 处理 ETL 和数据集成；
- 用 SQL、Python、配置、脚本和平台工具把客户现实接入 Palantir 平台。

所以这篇的“例子”不是某一个业务对象，而是一个交付动作：

```text
把客户现场的模糊业务问题，翻译成平台上的数据模型、配置、权限、工作流和可运行界面。
```

## 这篇和当前主线的关系

我们当前主线是：

```text
ontology = 基于表数据的面向对象业务表达
业务对象 + 关系 + action + workflow = 可运行的业务系统表达
```

FDE 这篇补的不是“ontology 是什么”，而是“这种东西怎么进入真实客户现场”。

一个更稳的理解是：

```text
Palantir 的价值不是只有平台，也不是只有人力服务。
它像是：强平台 + 前线工程实施 + 客户现场问题翻译。
```

如果没有 FDE 这种角色，复杂平台可能很难被客户自助使用；但如果没有 Foundry / Gotham 这种底层平台，
FDE 又可能只是普通实施顾问。

## 文章中的核心判断

文章倾向于认为：

```text
FDE 更像一种品牌化的高级企业软件实施角色，
而不是一种全新的软件工程革命。
```

它把 FDE 拆成几个传统角色的组合：

- 高端 IT 咨询；
- 专业实施服务；
- 售前 / 售后技术工程；
- 大客户技术支持；
- 数据工程和脚本胶水工作；
- 现场业务翻译和交付负责人。

文章也承认 FDE 有真实价值：反馈快、离客户近、能理解业务、能把平台能力落地。
但它反对把 FDE 神化成纯粹的核心研发或全新职业物种。

## 最值得吸收的地方

这篇对 wiki 有价值的点是：

```text
ontology / AIP 的业务价值不是只靠模型定义产生的，
还依赖把客户现实翻译进平台的人和交付机制。
```

这能补上我们之前阅读里的一个缺口：

```text
为什么 Palantir 这种高度抽象的平台没有变成普通低代码平台？
```

一个可能答案是：

```text
它不是单纯卖一个可以自助使用的软件，
而是用 FDE 把平台能力嵌入客户现场，
让客户的问题在平台抽象里被重新表达和运行。
```

## 需要警惕的地方

这篇本身是二手评论，且带有明显“去魅”立场。它可以帮助我们抵抗营销叙事，但不能单独证明 FDE 的真实工作比例。

它不能证明：

- Palantir FDE 真的主要做配置而不是研发；
- FDE 与传统咨询没有本质差异；
- Palantir 平台能力是否被夸大或低估；
- FDE 模式在所有客户和行业里都一样；
- FDE 模式可以脱离 Foundry / Gotham 被复制。

所以更稳的吸收方式是把它作为一个问题框架：

```text
Palantir 的价值到底来自产品平台，还是来自高强度现场实施？
两者如何组合，才形成客户感知到的能力？
```

## Review 判断

这篇不适合作为 Palantir 产品能力的直接证据，但适合作为“交付模型”的候选材料。

如果要进入正式 wiki，我建议吸收成一个比较克制的判断：

```text
Palantir 的 ontology/AIP 能力需要通过交付机制进入客户现实。
FDE 可以被理解为平台能力与客户业务之间的前线翻译和实施角色。
它的价值同时依赖平台抽象能力和现场工程劳动。
```

这个判断比“FDE 是革命”或“FDE 只是营销”都更有用。

## 和当前工作理解的关系

当前我们对 Palantir 主线的理解更像：

```text
源系统表数据
-> ontology 对象、属性、关系
-> action / workflow
-> 业务界面与 AI 判断
-> 人或自动化机制执行
```

FDE 补的是另一条横向能力：

```text
客户现实
-> 业务问题识别
-> 数据接入和清洗
-> ontology / workflow / 权限配置
-> 可运行解决方案
-> 现场反馈和迭代
```

也就是说，FDE 不是 ontology 里的对象，也不是 action 本身。它是让这些抽象能在客户环境中运行起来的交付角色。
