# Review：Palantir 揭秘：软件开发的终局，是被“本体”统一？

## 当前状态

待讨论。

这篇重新回到 ontology 主线，但它讨论的不是“本体如何建第一个对象”，而是：

```text
如果企业已有很多数据库、API、SaaS 和内部系统，
开发者能否不再为每个应用重复写胶水代码，
而是通过共享 ontology 和 OSDK 直接操作业务对象？
```

它是 `Ontology-Oriented Software Development` 的二手解读。

## 先讲清楚：这篇到底举了什么例子

文章举了一个航空公司预防性维护 / 航班调换场景。

一架飞机预检警示灯亮了。业务上要判断：

```text
这架飞机还能不能飞？
是否需要维修？
哪个机场有备件和认证维修师？
如果换飞机，哪个替代方案对旅客影响最小？
是否需要审批，还是可以自动执行？
```

这需要跨很多系统取数据和做动作：

```text
飞机 IoT 数据
备件库存
维修人员排班
航班计划
旅客影响
风险模型
运营系统写回
通知和审批
```

文章把它拆成两种开发方式。

### 传统组件/API 开发方式

开发者要写大量胶水代码：

```text
连接 MySQL
写 SQL 查库存和航班
调用 REST API 计算调换方案
调用 REST API 计算影响成本
配置认证 token
调用 swapAircraft API
配置邮件服务器发审批邮件
处理各种 JSON 和错误
```

文章要表达的点是：

```text
代码的大部分不是业务逻辑，而是在处理系统连接、数据结构、认证和 API 细节。
```

而这些连接知识被锁死在单个应用里，其他团队难以复用。

### 本体导向开发方式

在本体导向开发里，系统连接、数据模型、业务关系和部分逻辑已经沉淀到共享 ontology。

开发者不再直接面对：

```text
数据库表
字段名
REST endpoint
认证细节
孤立 JSON
```

而是直接面对业务对象：

```text
Aircraft
Flight
Airport
Part
MaintenanceSchedule
SwapOption
```

文章将 OSDK 描述成一种让开发者“用业务语言构建 API”的工具。

## 这篇和我们当前理解的关系

你刚才的理解可以作为这篇的关键读法：

```text
Palantir 通过数据库、API、文件、数据管道、连接器等方式适配客户现有系统；
原有系统不一定被替换，仍可作为事实来源和执行系统；
Palantir 在上层建立 ontology，把数据和动作映射成业务对象、关系、workflow 和 action；
开发者和业务人员可以在 Palantir 上做更多业务工作；
是否写回原系统，取决于 action、权限、审批、审计和回滚边界。
```

这篇支持这个方向，但从开发者体验角度讲：

```text
ontology 的价值之一，是把系统集成知识从单个应用代码里提升为共享业务 API。
```

也就是说，Palantir 不是简单把 SAP / 数据库 / API 接起来，而是试图把这些连接后的业务语义沉淀成一层可复用的对象模型。

## 最值得吸收的判断

这篇最有价值的判断是：

```text
企业软件开发的问题不只是组件能力不足，而是跨组件、跨系统的业务语义和操作边界没有统一沉淀。
```

Ontology-oriented development 的目标不是替代所有底层系统，而是：

```text
把已有系统中的数据、逻辑和动作映射成共享业务对象层；
让应用、低代码工具、AI 和人工用户都围绕同一套业务对象工作。
```

这与我们之前的理解一致：

```text
原系统 = 事实来源和执行系统
Palantir = 数据治理 + 业务建模 + AI / workflow / action 层
```

## 和 AI 的关系

文章还提出一个判断：

```text
企业 operational AI 的关键不只是模型能力，而是有没有稳定的企业世界模型。
```

它认为 AI 不看漂亮 UI，而需要结构化 API。ontology 可以同时提供三类入口：

```text
GUI：给人用；
API / OSDK：给程序员和 AI 系统用；
NLI：给自然语言交互用。
```

这个判断和我们前面的 OAG / RAG 讨论相通：AI 如果只面对文档和接口，很难稳定理解业务；如果面对治理后的对象、关系、action 和权限，才更可能成为企业运行系统的一部分。

## 需要警惕的地方

这篇宏大叙事很强。

标题里的“软件开发的终局，是被本体统一”不能照收。更稳妥的理解是：

```text
ontology 是一种当前有用的开发和业务抽象模式，类似面向对象、领域模型或共享业务 API。
模式总会随着业务、工具和组织变化而演化，不应被写成终局。
```

它不能直接证明：

- Palantir 的 ontology 已经真的统一了软件开发；
- OSDK 可以充分屏蔽所有底层系统复杂度；
- 定制开发边际成本可以接近零；
- ontology 层不会产生新的复杂性、迁移成本和治理负担；
- AI 只要有 ontology 就能可靠执行复杂业务动作。

更稳妥的读法是：

```text
这篇提供的是 ontology-oriented development 的产品愿景和开发者体验叙事。
它对我们有用，但需要后续 OSDK、action、权限、写回和版本治理材料验证。
```

## Review 判断

这篇值得进入正式 wiki，但应该作为“开发者体验 / 业务 API 层”的材料吸收。

建议吸收的判断：

```text
ontology 的一个重要价值，是把客户现有系统的数据、逻辑和动作映射成共享业务对象层，
减少每个应用重复处理数据库/API/认证/字段含义的胶水代码。
```

不建议吸收的判断：

```text
软件开发终局已经被 ontology 统一；
OSDK 已经消除了集成复杂性；
企业 AI 问题完全不是 AI 问题。
```

用户判断：

```text
这篇标题有问题。本体只是一种模式，类似面向对象；模式总在变，不能当作软件开发终局。
```

进一步的用户判断：

```text
本质上仍然都是开发。过去是每个具体需求都要由需求者口述，再由开发者理解并写代码；
现在多了一层中间件/平台抽象，通过构建模型、关系和 action，让需求面向平台或工具表达。
这减少了一级临时理解，但增加了一级抽象。它有好有坏，好坏取决于具体需求。
```
