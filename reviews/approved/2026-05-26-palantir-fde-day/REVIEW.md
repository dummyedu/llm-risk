```text
读患者指南
-> 理解数据信任和治理边界
-> 整合医院 ICU 数据与社区医生随访记录
-> 用 Foundry 处理数据血缘和权限
-> 给护士长做 Workshop
-> 把多系统查询变成一个业务界面
```

## 先讲清楚：这篇到底举了什么例子

文章核心例子是英国 NHS 的医疗数据平台场景。

它先说 FDE 的一天不是从写代码开始，而是从阅读《Foundry 患者指南》开始。这个细节很重要，因为医疗数据场景里，技术问题首先是信任和治理问题：

```text
谁控制数据？
谁只是处理数据？
谁可以访问？
访问是否留下记录？
患者是否能理解平台如何使用数据？
```

文章给出的边界是：

```text
NHS 是数据控制者。
Palantir 是数据处理者。
Palantir 只能按 NHS 规则处理数据，不能把数据用于其他目的。
```

随后文章给出一个技术任务：

```text
把 A 医院的重症监护数据，与 B 社区的家庭医生随访记录安全整合。
```

这不是简单 ETL，因为两边的数据模型、标准和命名法不同，还涉及隐私、权限和数据血缘。

文章最后给出一个用户赋能场景：

```text
FDE 给护士长做 Workshop。
护士过去需要登录七八个系统、打很多电话，才能确认手术排期和床位信息。
FDE 通过 Foundry 界面展示手术室实时安排、病人术前准备和病房床位占用。
```

这个例子比“FDE 很会沟通”更有价值，因为它展示了一个从数据整合到业务界面的现场交付链条。

## 这篇和上一篇 FDE review 的关系

上一篇 FDE 文章帮助我们形成了一个判断：

```text
FDE 是平台能力与客户现实之间的前线翻译和实施角色。
```

这篇可以给这个判断补一个更具体的工作切面：

```text
FDE 的现场建模不是只画对象图。
它还包括信任边界、数据治理、用户工作流和现场培训。
```

也就是说，FDE 建模的对象不只是业务实体，也包括：

- 数据责任边界：谁是 controller，谁是 processor；
- 数据来源和血缘：数据从哪里来，经过什么处理；
- 权限边界：谁能看，谁能操作；
- 用户工作流：护士长、医生、管理员如何在界面里完成任务；
- 信任表达：患者和一线用户是否能理解系统如何使用数据。

## 对我们当前理解的补充

此前我们讨论 FDE 时，重点在：

```text
围绕客户问题建立局部业务模型，而不是建立所有模型。
```

这篇进一步提示：

```text
在高敏感行业里，局部模型不能只包含业务对象和 action。
它还必须包含数据治理、权限、血缘和用户信任。
```

比如 NHS 例子里，只建这些对象不够：

```text
Patient
ICU Record
GP Follow-up
Surgery Schedule
Bed Occupancy
```

还必须回答：

```text
NHS 和 Palantir 的数据责任关系是什么？
谁可以看患者数据？
每次访问是否可审计？
数据如何从不同系统汇入？
医护人员如何在一个界面里完成原本跨系统的工作？
患者如何知道自己的数据被如何使用？
```

所以这篇对建模能力的启发是：建模边界不只由业务 action 决定，也由信任、权限、审计和用户采用决定。

## 需要警惕的地方

这篇营销感很强，且有“FDE 一天”的文学化叙事。它不能单独证明：

- NHS 项目的真实实施细节；
- 某个具体 FDE 的真实日程；
- Foundry 在 NHS 的实际效果；
- “每天节省两个小时”这类效果数字的可靠性；
- Palantir 与 NHS 的争议已经被患者指南完全解决。

更稳妥的读法是：

```text
把它当作 FDE 工作内容的叙事样本，而不是强证据。
```
