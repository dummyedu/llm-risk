# Palantir FDE 的一天

## 这篇在讲什么

这篇文章用“FDE 的一天”来描述 Palantir 前线部署工程师的现场工作。它不是严格案例复盘，
也不是产品教程，而是一篇带有宣传色彩的角色叙事。

它对本 wiki 的价值是表征性的：帮助理解 FDE 的主要价值和客户现场的大致情况。

```text
FDE 的价值不是只写代码，
而是在客户现场把业务问题、数据治理、平台配置和一线工作流连接起来。
```

## NHS 场景

文章最具体的场景是英国 NHS 医疗数据平台。

它说 FDE 的一天不是从写代码开始，而是从阅读《Foundry 患者指南》开始。这个细节说明，
在医疗这种高敏感数据场景里，技术交付首先要面对信任和治理问题：

```text
谁控制患者数据？
谁只是处理数据？
谁可以访问？
访问是否留下记录？
患者是否能理解数据如何被使用？
```

文章给出的表述是：

```text
NHS 是数据控制者。
Palantir 是数据处理者。
```

这不能证明 NHS 项目争议已经解决，但它能说明 FDE 现场工作中会遇到数据责任、权限、血缘和用户信任这些问题。

## 现场工作切片

文章举出的任务是：

```text
把 A 医院的重症监护数据，与 B 社区的家庭医生随访记录安全整合。
```

这表面是 ETL，实际会涉及不同数据模型、标准、命名法、权限规则和隐私要求。

文章还讲到 FDE 给护士长做 Workshop，让一线人员在一个界面里查看手术室安排、术前准备和床位占用，而不是登录多个系统、反复打电话确认信息。

这个例子的意义是：

```text
FDE 的交付终点不是“数据接入完成”，
而是一线人员能在可信界面里完成原来的复杂工作。
```

## 不能过度吸收

这篇不能证明：

- NHS 项目的真实效果；
- FDE 的真实日程就是文章所写；
- Foundry 已经充分解决医疗数据治理问题；
- “节省时间”等效果数字可靠。

因此它只作为 FDE 交付模型的表征性材料使用。
