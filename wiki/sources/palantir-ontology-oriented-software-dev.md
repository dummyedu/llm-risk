# Palantir 本体导向软件开发

## 这篇在讲什么

这篇文章讨论 `Ontology-Oriented Software Development`，但标题里的“软件开发终局”不应照收。

更稳妥的读法是：

```text
ontology 是一种当前有用的开发和业务抽象模式，
类似面向对象、领域模型或共享业务 API。
它不是软件开发终局，模式会随着业务、工具和组织变化而演化。
```

## 核心例子

文章用航空公司预防性维护 / 航班调换场景说明问题。

飞机预检警示灯亮起后，业务需要判断：

```text
飞机还能不能飞？
是否需要维修？
哪个机场有备件和维修师？
如果换飞机，哪个替代方案影响最小？
是否需要审批，还是可以自动执行？
```

传统开发方式会让开发者写大量胶水代码：

```text
连接数据库
写 SQL
调用 REST API
处理认证 token
处理 JSON
发邮件审批
调用底层 swap 操作
```

文章要表达的不是“这些代码不存在了”，而是：

```text
这些系统连接、字段含义、权限和操作边界，如果每个应用都重新理解，会造成大量重复开发。
```

## 更有用的理解

本体导向开发的价值，不是自动完成代码，而是改变开发组织方式。

过去更像：

```text
每个需求
-> 业务口述
-> 开发临时理解底层系统、字段、API、权限
-> 写具体功能代码
```

现在它希望变成：

```text
先开发共享 ontology / 业务对象层
-> 把稳定对象、关系、权限和 action 放进去
-> 后续需求尽量面向这层平台表达
-> 应用、低代码、AI 和 workflow 在这层之上组合
```

所以 ontology 本身也是开发：

```text
对象建模
关系连接
字段和 API 映射
action 绑定
权限和审计设计
性能判断
版本治理
```

它减少了每个需求中的临时理解和胶水代码，但增加了一层抽象的设计、维护和治理成本。好坏取决于需求类型、复用度和风险。

## 对综合页的影响

这篇已吸收到 [[concepts/ontology]]。

吸收的判断是：

```text
ontology 的一个价值，是把客户现有系统的数据、逻辑和动作映射成共享业务对象层，
让应用、低代码工具、AI 和人工用户围绕同一套业务对象工作。
```

不吸收的判断是：

```text
软件开发终局已经被 ontology 统一；
OSDK 已经消除了所有集成复杂性；
企业 AI 问题完全不是 AI 问题。
```

## 来源与追溯

- 来源事实：文章以航空公司飞机预检警示灯为例，说明一次决策需要 IoT、库存、维修排班、航班、旅客影响模型和运营写回等多系统协作。
- 来源事实：文章用传统 Node.js 示例说明传统开发需要连接数据库、调用 REST API、处理认证、JSON、邮件审批和调换操作。
- 来源事实：文章称本体导向开发中，开发者通过 OSDK 直接与飞机、航班计划、机场等业务概念互动。
- 用户判断：本体只是一种模式，类似面向对象；模式总在变，不能当作软件开发终局。
- 用户判断：本质上仍然都是开发；ontology 减少一级临时理解，但增加一级抽象，好坏取决于具体需求。
