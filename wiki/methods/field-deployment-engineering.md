# FDE 与现场交付

## 是什么

FDE（Forward Deployed Engineer）在当前 wiki 中不是一个单纯岗位介绍，而是一种企业 AI
平台进入客户现场的交付方法。

它连接三件事：

```text
客户现场问题
-> 平台里的业务对象、权限、工作流、界面和 action
-> 可运行、可治理、可继续沉淀的平台能力
```

因此，FDE 不应被理解为“写代码的人”或“咨询顾问”的单一角色。更合适的阶段性理解是：

```text
平台能力与客户现实之间的前线翻译和实施角色。
```

## 核心能力是建模

FDE 的关键能力不是先知道某个工具菜单在哪里，而是能把客户的模糊问题拆成可运行结构。

客户通常不会说“我要一个 ontology”。客户说的是：

```text
订单处理太慢
供应链风险看不清
审批流程反复靠人催
一线人员要在多个系统之间来回切换
```

FDE 要把这些话翻译成：

```text
关键对象
-> 关键属性
-> 对象关系
-> 状态变化
-> action
-> workflow
-> 权限、审批、审计
-> 用户界面和反馈
```

这就是 FDE 和 [[methods/ontology-centered-delivery]] 的关系：FDE 是这种交付方法在客户现场的执行者之一。

## 不建立完整模型

FDE 不应该试图建立完整企业模型。完整模型会无限展开，既拖慢交付，也可能脱离当前业务价值。

更合适的边界是：

```text
只建模到足以支撑当前判断、动作、协作和追责的程度。
```

也就是说，模型服务于当前业务闭环，而不是服务于“把企业一次性描述完整”。

可以用四层判断建模深度：

```text
记录层：只是存下来，方便查。
解释层：作为人或 AI 的上下文，不直接触发动作。
决策层：参与判断、评分或推荐。
执行层：触发 action，改变业务系统，必须有权限、审批、审计和回滚。
```

越靠近执行层，建模、权限和审计越要明确。

## 现场交付不是只接数据

FDE 的现场工作不只是 ETL 或 dashboard。客户现场经常同时包含：

```text
敏感数据
既有系统
权限边界
业务责任
用户习惯
组织信任
```

所以现场交付的真实任务更像：

```text
业务数据整合
-> 数据责任和访问边界
-> 可追溯的数据血缘
-> 一线用户能使用的工作界面
-> 现场培训和反馈
```

这解释了为什么一个企业 AI 平台不能只靠“模型能力”或“产品功能列表”完成交付。它还需要有人把平台能力翻译成客户真实工作。

## 平台可交付性是底座

FDE 能工作，前提是平台本身能在客户环境里可靠运行。

对 Palantir 这类企业平台，交付还包括：

```text
商业云、私有云、本地机房和隔离网络部署
持续升级和灰度发布
监控、告警和远程支持
机密网络中的前线代理
合规环境建设
边缘设备上的受限部署
```

这类能力不是 FDE 的替代，而是 FDE 能把业务模型落地的运行底座。

## 如何沉淀成产品能力

FDE 模式最大的风险是长期依赖高强度人力服务。

要避免这一点，现场经验必须反向沉淀为平台能力：

```text
一线部署痛点
-> 临时脚本
-> 自动化工具
-> 平台核心能力
```

如果客户现场的问题不能回流到平台，FDE 就更像高端实施服务；如果能持续回流，FDE 才可能成为平台进化机制的一部分。

## 当前判断

Palantir 的客户价值应暂时理解为：

```text
强平台
+ 前线工程实施
+ 客户现场问题翻译
+ 现场经验回流为平台能力
```

这不是单纯产品化，也不是单纯咨询服务，而是平台和现场交付互相推动的模式。
